# 交互式校园 WiFi 信誉奖惩仿真 Demo 开发文档

## 1. 文档目的

本文档用于指导开发一个可互动、可演示、可复现的校园 WiFi 共享信誉奖惩机制仿真 Demo。  
Demo 的目标不是复现真实 802.1X 或 RADIUS 网络，而是把论文中的机制、动态演化过程、基线对比和参数敏感性清晰展示出来，服务于答辩、汇报、课程展示和论文辅助说明。

Demo 需要回答四个核心问题：

1. 该信誉奖惩机制是如何运行的。
2. 它如何改变用户共享行为和系统收益。
3. 它相对基线方案是否提升了共享率、吞吐和公平性。
4. 在哪些参数区间或检测误差下，机制会失效或退化。

## 2. Demo 定位与范围

### 2.1 产品定位

这是一个“交互式仿真看板”，而不是一个真实校园网控制系统。

用户可以：

1. 调整模型参数并启动仿真。
2. 观察网络中用户策略和信誉随时间变化。
3. 对比所提方案与基线方案的表现。
4. 批量扫描参数，查看鲁棒性和敏感性。
5. 导出配置、图表和结果日志，保证实验可复现。

### 2.2 MVP 边界

首版 Demo 不做以下内容：

1. 真实校园地图接入。
2. 真实 AP 遥测数据接入。
3. 真实认证、RADIUS、CoA 或控制平面联动。
4. 真实带宽压测。

首版必须保留以下约束：

1. 未给出的物理量标记为“未指定”或“归一化”。
2. 所有基线方案使用同一组参数基础，保证比较公平。
3. 当重复次数 `R > 1` 时，结果必须展示 `mean +/- 95% CI`。
4. `Replicator` 与 `Moran` 两类仿真结果必须分开展示，不能混为同一视图。

## 3. 目标用户与使用场景

### 3.1 目标用户

1. 论文作者本人，用于答辩展示。
2. 导师或评审，用于快速理解机制和实验设置。
3. 课程展示观众，用于直观看到参数变化带来的行为差异。

### 3.2 核心使用场景

1. 单次动态演示：展示一个参数配置下，系统如何从初始状态演化到稳定状态。
2. 基线对比演示：切换方案后，对比共享率、吞吐、公平性和稳定时间。
3. 参数扫描演示：批量运行后展示热图，说明方法的适用范围和失效边界。
4. 复现实验演示：导出配置文件和原始日志，证明实验可复现。

## 4. 业务目标

### 4.1 必达目标

1. 用户能在 1 分钟内理解机制输入、过程和输出。
2. 用户能在 3 分钟内完成一次“参数调整 -> 运行 -> 对比 -> 导出”的闭环。
3. Demo 能清楚展示“为什么该机制优于基线”，而不是只给结论。
4. 所有实验结果可通过固定 `seed`、`R`、`T` 和配置文件复现。

### 4.2 成功标准

1. 用户能从界面直接看出共享率 `x(t)` 的变化趋势。
2. 用户能直接比较 Proposed、No Reputation、Fixed Reward、Random Strategy 四类方案。
3. 用户能直接观察 `theta`、`alpha`、`kappa`、`lambda`、`gamma`、`p_d`、`p_f` 对结果的影响。
4. 所有图表和表格具备清晰坐标轴、图例、单位或“归一化/未指定”说明。

## 5. 核心概念与模型约束

### 5.1 场景抽象

校园网络使用图模型 `G = (V, E)` 表示，其中：

1. 节点表示建筑、区域或用户群。
2. 边表示覆盖、接入或可共享关系。
3. AP 数为 `M`，用户数为 `N`，初始共享比例为 `x0`。

若无真实地图，使用随机几何图或网格图近似。  
真实坐标、覆盖半径、物理层带宽、真实业务到达率、AP 容量若无来源，一律标记为“未指定”，并使用归一化容量表达。

### 5.2 支持的方案

Demo 至少支持以下方案：

1. `Proposed`：信誉阈值奖惩机制。
2. `No Reputation`：`R = 0, P = 0`。
3. `Fixed Reward`：对共享者给予固定补贴，且 `P = 0`。
4. `Random Strategy`：策略按固定概率更新，不依赖收益或信誉。

### 5.3 支持的仿真引擎

首版至少支持：

1. `Replicator`：主仿真模式，用于连续时间或离散步的演化过程展示。

第二阶段建议支持：

1. `Moran`：用于有限群体随机演化验证，输出 `rho_S` 和吸收时间。

## 6. 交互式 Demo 的功能模块

### 6.1 机制概览模块

用于用最短路径解释机制本身。

展示内容：

1. 机制流程图。
2. 用户状态说明：共享者、非共享者、被奖励、被惩罚。
3. 奖励/惩罚触发逻辑。
4. 信誉阈值 `theta` 的作用解释。

交互要求：

1. 鼠标悬停参数时显示定义和影响。
2. 支持展开“机制说明”面板。

### 6.2 参数控制模块

用于设置仿真输入。

必填参数：

1. `scheme`
2. `engine`
3. `N`
4. `M`
5. `x0`
6. `theta`
7. `alpha`
8. `kappa`
9. `lambda`
10. `gamma`
11. `p_d`
12. `p_f`
13. `T`
14. `R`
15. `seed`

交互要求：

1. 支持滑块、数字输入框和下拉选择。
2. 支持“恢复默认演示参数”。
3. 支持导入/导出 JSON 或 YAML 配置。
4. 参数越界时给出即时校验提示。

### 6.3 动态演化模块

这是 Demo 的核心演示区域。

展示内容：

1. 拓扑示意图或网格图。
2. 节点颜色表示策略状态。
3. 节点边框或透明度表示信誉强弱。
4. 时间轴驱动下的动态变化。

交互要求：

1. 支持 `播放 / 暂停 / 单步 / 重置 / 调速`。
2. 支持拖动时间轴跳转到任意步 `t`。
3. 点击节点后显示该节点的策略、信誉、收益和历史曲线。

### 6.4 动态指标模块

用于显示系统级指标随时间变化。

必须展示：

1. `x(t)`：共享率时间序列。
2. `Th(t)`：系统吞吐时间序列。
3. `J(t)`：Jain 公平性时间序列。
4. `mean(r_t)`：平均信誉时间序列。

建议展示：

1. sharer count
2. defector count
3. rewarded-user count
4. punished-user count

交互要求：

1. 支持多曲线联动高亮。
2. 支持悬停显示当前时刻所有指标值。
3. 支持同步跟随时间轴移动。

### 6.5 稳态与分布模块

用于展示仿真收敛后的结果。

必须展示：

1. 稳态共享率 `x*`
2. 平均吞吐 `Th`
3. 平均公平性 `J`
4. 稳定时间 `T_s`
5. 收益分布 `P10 / P50 / P90`

建议展示：

1. 最终信誉分布直方图
2. 最终策略组成饼图或堆叠柱图

### 6.6 基线对比模块

用于比较不同方案的最终效果。

必须对比以下方案：

1. Proposed
2. No Reputation
3. Fixed Reward
4. Random Strategy

展示形式：

1. 对比表
2. 分组柱状图
3. 误差线图

对比指标：

1. `x*`
2. `Th`
3. `J`
4. `T_s`
5. `P10 / P50 / P90`

要求：

1. 同一组参数下比较所有方案。
2. 当 `R > 1` 时显示均值和置信区间。

### 6.7 敏感性与鲁棒性模块

用于展示机制在哪些参数下表现更好，在哪些参数下会退化。

必做扫描参数：

1. `theta`
2. `alpha`
3. `kappa`
4. `p_f`
5. `p_d`

建议一起支持：

1. `lambda`
2. `gamma`

建议图表：

1. `alpha-kappa` 热图，对应 `x*`
2. `alpha-kappa` 热图，对应 `J`
3. `alpha-kappa` 热图，对应 `T_s`
4. `p_f-p_d` 鲁棒性热图

参数扫描范围按实验设计文档执行：

1. `theta in [0.4, 0.9]`
2. `alpha in [0, 1]`
3. `kappa in [0, 1]`
4. `lambda in [0.05, 0.5]`
5. `gamma in [0, 0.5]`
6. `p_d in [0.6, 0.99]`
7. `p_f in [0.01, 0.2]`

### 6.8 复现与导出模块

用于保证实验严谨性和可复现性。

必须支持：

1. 导出当前配置文件 `JSON/YAML`
2. 导出每步日志 `CSV`
3. 导出每次运行摘要 `CSV/JSON`
4. 导出图表 `PNG/SVG`

必须包含的元数据：

1. `seed`
2. `R`
3. `T`
4. `scheme`
5. `engine`
6. 时间戳
7. 参数快照

## 8. 技术方案

### 8.1 推荐技术栈

前端建议：

1. `React + TypeScript + Vite`
2. 状态管理使用 `Zustand`
3. 图表使用 `ECharts` 或 `Plotly`
4. 样式使用 `Tailwind CSS` 或 `CSS Modules`
5. 动画使用 `Framer Motion`
6. 大批量仿真计算放入 `Web Worker`

理由：

1. 易于快速搭建交互看板。
2. 前端本地即可运行，部署成本低。
3. 计算和渲染可以分离，避免界面卡顿。

### 8.2 推荐架构

采用“前端主线程 + 仿真 Worker”的双层架构。

模块划分：

1. `UI Shell`：页面布局、导航、页签。
2. `Config Store`：当前参数、预设、校验状态。
3. `Simulation Worker`：执行单次仿真和批量扫描。
4. `Result Store`：缓存时间序列、统计摘要和热图矩阵。
5. `Chart Layer`：负责曲线、热图和比较图渲染。
6. `Export Service`：配置、日志和图表导出。

### 8.3 是否需要后端

MVP 阶段不强制需要后端。  
若后续需要多人共享、任务队列、长时间批量扫描或数据库持久化，再增加后端服务。

## 9. 仿真引擎输出契约

### 9.1 输入配置格式

推荐使用 JSON 或 YAML。

示例：

```yaml
scheme: Proposed
engine: Replicator
N: <demo_default>
M: <demo_default>
x0: <demo_default>
theta: 0.7
alpha: 0.5
kappa: 0.4
lambda: 0.2
gamma: 0.1
p_d: 0.9
p_f: 0.05
T: 2000
R: 30
seed: 20260318
```

说明：

1. `N`、`M`、`x0` 可由产品预设默认值填充，但必须标记为“演示默认值”，不能冒充论文实测值。
2. 未给出的真实物理单位不做虚构补充。

### 9.2 每步日志格式

每步至少输出以下字段：

1. `t`
2. `x(t)`
3. `Th(t)`
4. `J(t)`
5. `mean(r_t)`
6. `sharer_count`
7. `defector_count`
8. `rewarded_user_count`
9. `punished_user_count`

若界面支持节点详情，可扩展输出：

1. `node_id`
2. `strategy`
3. `reputation`
4. `utility`

### 9.3 单次运行摘要格式

每次运行至少输出：

1. `x_star`
2. `mean_Th`
3. `mean_J`
4. `T_s`
5. `utility_P10`
6. `utility_P50`
7. `utility_P90`
8. `final_reputation_histogram`

`Moran` 模式下额外输出：

1. `rho_S`
2. `mean_absorption_time`

### 10.2 图表交互要求

2. 时间序列图支持与时间轴联动。
3. 热图支持点击单元格回填参数到控制区。

1. 坐标轴必须有名称。
2. 单位缺失时明确写“normalized”或“未指定”。
3. 置信区间必须使用误差线或阴影带表示。
4. 不允许只给图，不解释当前配置和方案名称。

## 11. 动画与交互细节

### 11.1 时间控制

必须支持：

1. `Play`
2. `Pause`
3. `Step`
4. `Reset`
5. `Speed x1 / x2 / x4`
6. 拖动进度条跳转

### 11.2 节点动画

建议约定：

1. 共享者用暖色。
2. 非共享者用冷色。
3. 信誉值通过光晕、透明度或边框粗细表达。
4. 奖励事件和惩罚事件使用短暂闪烁或脉冲动画。

### 11.3 讲解辅助

建议提供：

1. “当前发生了什么”文本提示区。
2. 关键时刻自动标注，如共享率快速上升、趋稳、失稳。
3. “一键加载典型案例”按钮。

## 12. 实验矩阵设计

### 12.1 单次演示实验

目的：让观众理解机制运行过程。

要求：

1. 固定一组典型参数。
2. 用动画 + 曲线同时展示从 `t = 0` 到稳定的过程。
3. 显示关键指标卡片实时更新。

### 12.2 基线对比实验

目的：证明方法优于基线。

要求：

1. 对 Proposed、No Reputation、Fixed Reward、Random Strategy 使用同一参数配置。
2. 每个方案至少运行 `R = 30` 次。
3. 汇总 `x*`、`Th`、`J`、`T_s`、`P10/P50/P90`。

### 12.3 参数敏感性实验

目的：说明机制在哪些条件下更有效。

要求：

1. 扫描 `theta`、`alpha`、`kappa`、`p_f`、`p_d`。
2. 至少展示热图或折线族图。
3. 强调高误检、低检测率、极端阈值时的退化情况。

### 12.4 Moran 验证实验

目的：说明有限群体随机演化下的稳健性。

要求：

1. 独立于 `Replicator` 展示。
2. 展示 `rho_S` 和吸收时间。
3. 不与主动态演化图混用同一图层。

## 13. 性能与工程要求

### 13.1 性能要求

1. 单次演示在常规笔记本上应流畅运行。
2. 批量扫描任务不能阻塞 UI 主线程。
3. 大于 1 万步的可视化回放应支持抽样或分帧渲染。

### 13.2 工程要求

1. 参数变更、运行结果、导出文件必须可追踪。
2. 图表、控制器、仿真引擎应模块化。
3. 仿真算法层与 UI 层分离。
4. 所有默认值、预设和实验网格可配置。

## 16. 验收标准

以下条件同时满足，视为 Demo 达到首版验收标准：

1. 用户可在界面中修改参数并启动仿真。
2. 系统能展示动态拓扑或状态动画。
3. 系统能同步展示 `x(t)`、`Th(t)`、`J(t)`、`mean(r_t)`。
4. 系统能切换四类方案并公平比较。
5. 系统能输出 `x*`、`Th`、`J`、`T_s`、`P10/P50/P90`。
6. 系统能完成参数扫描并生成至少一张热图。
7. 系统可导出配置、结果和图表。
8. 所有“未指定”物理参数均被明确标注，没有伪造真实值。

## 17. 风险与注意事项

1. 若仿真模型本身尚未完全定稿，先锁定输入输出契约，再迭代内部公式实现。
2. 若批量扫描计算量过大，优先保证单次演示流畅，再做异步批处理。
3. 若真实拓扑数据缺失，不要阻塞开发，应先用随机几何图或网格图替代。
4. 若 `Moran` 模式实现成本过高，可延期到第二阶段，但接口要预留。
